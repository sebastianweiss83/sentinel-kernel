# v3.5 Item 2 — Long-Term Semantic Interpretability (JSON-LD + PROV-O)

**Design-partner architectural point 2:** evidence packs retained for
10–15 years outlive the Sentinel codebase that produced them. By 2040
a regulator auditing a 2026 decision will not have our NDJSON schema
docs, probably not our Python SDK, and quite possibly not our domain.
Semantic export in a **standards-based** format — JSON-LD grounded in
the W3C **PROV-O** ontology — ensures the trace is still intelligible.

**Status:** Accepted · 2026-04-22 · ships in v3.5.0

---

## Problem

Sentinel's current NDJSON export is a snapshot of the *library's*
internal schema. Fields like `policy_evaluation.rule_triggered` or
`sovereignty.sovereign_scope` are meaningful only with access to the
Sentinel docs of the same release. For short-term evidence pack
distribution (audit running this quarter) that's fine. For long-term
retention (evidence pack read in 2040 by a successor auditor) the
schema is orphaned.

Two fixes:

1. **Emit the trace in a semantic web format** where each property
   carries its own IRI that resolves to documentation.
2. **Reuse an existing, widely-adopted ontology** so third-party
   tooling (W3C SHACL validators, JSON-LD processors in the Solid
   and broader semantic-web ecosystem, any PROV-aware
   provenance-store) can consume it without a Sentinel-specific
   plugin.

W3C **PROV-O** ([rec. 30 April 2013](https://www.w3.org/TR/prov-o/))
is exactly this kind of long-lived, stable, vendor-neutral ontology.
It models provenance — activities, agents, entities, their derivation
relationships — which is a near-perfect fit for decision traces.

---

## Scope

**In scope (v3.5.0 Item 2):**

- Add `comply.export(..., format="jsonld")` that produces a JSON-LD
  document mapping each trace to a `prov:Activity` with its inputs /
  outputs as `prov:Entity` instances and the running agent as a
  `prov:Agent`.
- Publish a **minimal Sentinel ontology** extending PROV-O at
  `https://sebastianweiss83.github.io/sentinel-kernel/ontology/v1/`
  — a single self-contained HTML page + `.jsonld` vocabulary file
  served from GitHub Pages. The vocabulary ships with the repo under
  `docs/ontology/v1/` and is published as part of the existing
  `Deploy to GitHub Pages` job.
- JSON-LD output is **validated** at generation time using `pyld`
  (expansion + compaction round-trip — guarantees the document is
  valid JSON-LD as shipped).
- Keep NDJSON as the default export format. JSON-LD is opt-in.

**Explicitly not in scope:**

- A full OWL-DL ontology. The v1 vocabulary is an ontology-as-HTML
  page with one `.jsonld` context file — enough to be de-referenceable
  IRIs, not a complete reasoning artefact.
- Round-tripping JSON-LD back into a `DecisionTrace`. One-way export
  only in v3.5.
- SHACL shapes. Deferred to v3.6 if a design partner asks.
- RDF/XML or N-Quads emission. JSON-LD is sufficient.

## Design

### Public surface

`sentinel/comply.py::export` gains a `format` kwarg:

```python
comply.export(sentinel, output, format="ndjson")   # default, existing behaviour
comply.export(sentinel, output, format="jsonld")   # new, v3.5
```

Calling `export()` without `format=` produces NDJSON as before — zero
behavior change for v3.4.x callers.

`format="jsonld"` writes a single JSON-LD document covering all traces
selected by the usual `EvidencePackOptions` (time range, project filter,
max_traces). The document is self-describing via its `@context`, which
references the published Sentinel vocabulary.

### Ontology layout

Published at `https://sebastianweiss83.github.io/sentinel-kernel/ontology/v1/`:

- `index.html` — human-readable vocabulary doc (one page, dark-mode
  V8-styled per CLAUDE.md design system)
- `context.jsonld` — JSON-LD `@context` object for Sentinel terms
- `sentinel.ttl` — the same vocabulary in Turtle for RDF consumers
  (generated once, hand-reviewed)

Term prefix: `sntl:` → `https://sebastianweiss83.github.io/sentinel-kernel/ontology/v1/#`

### Minimal Sentinel vocabulary (v1)

```
sntl:DecisionTrace     rdfs:subClassOf prov:Activity
sntl:PolicyEvaluation  rdfs:subClassOf prov:Activity
sntl:HumanOverride     rdfs:subClassOf prov:Activity
sntl:Agent             rdfs:subClassOf prov:SoftwareAgent
sntl:InputsHash        rdfs:subClassOf prov:Entity
sntl:OutputHash        rdfs:subClassOf prov:Entity

sntl:sovereignScope    rdfs:Property   # EU / LOCAL / CUSTOM
sntl:dataResidency     rdfs:Property
sntl:policyResult      rdfs:Property   # ALLOW / DENY / EXCEPTION
sntl:otelTraceId       rdfs:Property
sntl:schemaVersion     rdfs:Property
sntl:signature         rdfs:Property
sntl:signatureAlgorithm rdfs:Property
```

Everything else reuses PROV-O verbatim:

```
prov:Activity             — the DecisionTrace itself
prov:Agent / SoftwareAgent — the agent that ran
prov:Entity               — the input hash + output hash
prov:startedAtTime / endedAtTime — timing fields
prov:wasAssociatedWith    — activity ↔ agent link
prov:generated            — activity → output entity
prov:used                 — activity → input entity
prov:wasDerivedFrom       — output ↔ input link
```

### Trace-to-JSON-LD mapping

Each `DecisionTrace` becomes one JSON-LD node of type
`["sntl:DecisionTrace", "prov:Activity"]`. Sentinel-specific properties
live under `sntl:` terms; PROV-native properties (timestamps, activity
↔ agent links) use `prov:` terms.

Inputs and outputs become separate nodes of type
`["sntl:InputsHash", "prov:Entity"]` with `prov:value` being the SHA-256
hex digest. This lets any PROV-O-aware tool compute lineage queries
(`prov:wasDerivedFrom`) on hashes directly.

OTEL context fields from Item 1 are surfaced as
`sntl:otelTraceId`, `sntl:otelSpanId`, `sntl:otelParentSpanId`
properties on the activity.

### Why pyld

`pyld` is the canonical Python JSON-LD processor (digitalbazaar / JSON-LD
WG reference impl). Pure Python, MIT-licensed, no native deps, works
air-gapped (schema resolution happens against a local cache; we ship the
Sentinel `@context` inline in the document so no network fetch is needed
for validation).

Added as the `[jsonld]` optional extra. Emitting JSON-LD without the
extra raises an ImportError pointing at `pip install
'sentinel-kernel[jsonld]'`, same UX pattern as `[pdf]` for evidence
packs.

### Implementation layout

- `sentinel/comply/semantic.py` (new file) — pure-Python builders that
  convert a list of `DecisionTrace` + `EvidencePackOptions` into a
  JSON-LD dict. Uses `pyld` for canonical validation at the end.
- `sentinel/comply.py::export` — adds the `format` kwarg and
  delegates to the semantic builder.
- `docs/ontology/v1/` — `index.html`, `context.jsonld`, `sentinel.ttl`.
- `docs/ontology/v1/` is copied verbatim into `docs/preview/` by
  `scripts/sync_all.py` so GitHub Pages serves it.

### Round-trip validation (reality test)

The Phase 5 E2E test is: export an evidence pack as JSON-LD, feed the
document through `pyld.jsonld.expand()` (which resolves every IRI and
asserts every term is in a known context), then compact it back. If
the compacted form equals the original, the document is self-consistent
JSON-LD and every term is discoverable.

## Alternatives considered

**A. Roll our own XML or CSV export.** Rejected. Not interoperable
with any existing tool chain; becomes just another orphan schema.

**B. RDF/XML.** Rejected. Ergonomics are worse than JSON-LD and no
modern tool ecosystem beyond niche semantic-web stacks.

**C. Use DCAT or schema.org.** Rejected. DCAT is about *datasets*, not
activities; schema.org has no decision-provenance vocabulary.

**D. Target PROV-O directly without a Sentinel extension.** Tempting
but loses Sentinel-specific semantics (sovereignty scope, kill switch
state, signature algorithm). A thin subclass layer preserves
Sentinel's distinctiveness while staying PROV-O-compatible.

## Testing strategy

`tests/test_semantic_export_jsonld.py`:

1. `comply.export(..., format="jsonld")` produces a valid JSON-LD
   document — assert `@context` is present and points at the Sentinel
   vocabulary IRI.
2. Single trace export — verify one `sntl:DecisionTrace`/`prov:Activity`
   node with trace_id as `@id`, started_at as `prov:startedAtTime`.
3. Multi-trace export — each trace becomes a separate node; relationships
   preserved (`prov:wasDerivedFrom` between inputs and outputs where
   applicable).
4. OTEL context fields present when captured — `sntl:otelTraceId`
   populated.
5. Round-trip: `pyld.jsonld.expand()` → `pyld.jsonld.compact()`
   returns document equal to original.
6. Backward-compat: the existing NDJSON default path is unchanged.
7. Explicit unknown format raises clear ValueError.
8. `[jsonld]` extra missing → helpful ImportError.

Reality test (fresh venv, Phase 9):

```bash
pip install 'sentinel-kernel[jsonld]==3.5.0'
python -c "
from sentinel import Sentinel, comply

s = Sentinel(storage=':memory:', signer=None)
@s.trace
def credit(x): return {'approved': True, 'amount': x}
credit(1000)

comply.export(s, '/tmp/evidence.jsonld', format='jsonld')

import pyld, json
doc = json.load(open('/tmp/evidence.jsonld'))
expanded = pyld.jsonld.expand(doc)
# Assert PROV-O terms resolved
assert any('http://www.w3.org/ns/prov#Activity' in str(e) for e in expanded)
# Assert Sentinel terms resolved
assert any('sentinel-kernel/ontology/v1' in str(e) for e in expanded)
print('JSON-LD valid, PROV-O + Sentinel vocabulary both resolve')
"
```

## Rollout

- Ships in v3.5.0 alongside Items 1, 3, 4.
- Adds the `[jsonld]` optional extra. Core install size unchanged.
- Publishes the v1 ontology as part of the existing GitHub Pages
  deploy — no new infrastructure.
- `docs/preview/` adds an "Ontology" link in the header for external
  discoverability.
- Versioning: the ontology URL includes `/v1/`. v2 would live at
  `/ontology/v2/` so old evidence packs stay valid against the v1
  vocabulary.
