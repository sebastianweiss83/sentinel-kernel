# v3.5 Item 4 — Write-Once Storage Backend

**Berthold's fourth architectural point:** storage-layer tamper
**prevention**, not just detection. Sentinel already computes SHA-256
hashes and signs traces — a verifier can *detect* tampering after the
fact. But a regulated operator facing a DORA Art. 13 forensic
investigation or a BaFin §25a review needs the storage layer itself
to reject writes to already-attested traces. Detection after the
fact is not sufficient when the threat model includes a malicious
insider with write access to the SQLite file.

**Status:** Accepted · 2026-04-22 · ships in v3.5.0 (filesystem
backend); S3 Object Lock deferred to v3.6 (requires AWS credentials
for end-to-end testing that can't run in the existing CI matrix).

---

## Scope

**In scope (v3.5.0 Item 4):**

- A new `WriteOnceFilesystemStorage` backend that **rejects any save
  of a trace_id already on disk** at the application layer — the
  "append-only by convention" tier.
- Same backend applies the POSIX user-immutable flag (`chflags uchg`
  on macOS, `chattr +i` on Linux, best-effort) to the file after
  first write, so even direct filesystem writes by the owning user
  are rejected without `sudo chflags nouchg`. The best-effort flag
  is silently skipped on platforms or filesystems where it's not
  available — the software-level append-only guarantee still holds.
- A new optional field on `DecisionTrace`: `storage_mode` (default
  `"writeable"`, set to `"writeonce_fs"` when the backend is
  write-once filesystem). Evidence-pack consumers can see the
  storage discipline claimed for each trace.
- `Sentinel.verify_integrity` surfaces whether the trace's claimed
  storage mode is still honoured at read time (file still exists,
  still carries the immutable flag if applicable).

**Explicitly not in scope (deferred to v3.6):**

- S3 Object Lock backend. Design is anticipated in the backend
  interface (the `StorageBackend` contract already permits custom
  implementations) but its implementation and testing require live
  AWS credentials and a mocked `moto` layer worth its own phase.
- Azure Immutable Blob Storage. Same rationale.
- Fine-grained retention locks (per-trace policy-driven locking).
  The existing retention-policy module (v3.5 Item 3) captures the
  *intended* retention window; hardware-level lock enforcement is
  v3.6+.

## Design

### Backend interface extension

Today every backend subclasses `sentinel.storage.base.StorageBackend`.
We add **no** new abstract method — the write-once contract is a
runtime extension: backends that implement it declare themselves by
setting `backend_name = "writeonce_fs"` (or equivalent) and raising
`WriteOnceViolation` from `save()` when the trace_id already exists.

This choice keeps the existing `StorageBackend` contract stable for
every current and third-party backend. Write-once is *opt-in*, not a
new requirement.

### `WriteOnceViolation`

A new exception raised by write-once backends when `save()` is called
with a trace_id already present on disk:

```python
class WriteOnceViolation(RuntimeError):
    """Raised when a write-once backend rejects overwriting a stored trace.

    The trace_id that collided is the single argument.
    """
```

This is always an operator-level error — it either means a code bug
(two traces generated with the same uuid4, essentially impossible) or
a tampering attempt. Either way the application surface should
surface, not swallow.

### Filesystem layout

Same as `FilesystemStorage`: one NDJSON-lines file per trace at
`<root>/<trace_id>.ndjson`. On first write:

1. Write the file atomically (`write_bytes` to a `.tmp` + `os.replace`).
2. Apply the OS-level immutable flag (best effort).
3. Set `storage_mode="writeonce_fs"` on the trace before computing
   the signature so the signed payload reflects the claimed
   discipline.

On subsequent `save()` for the same trace_id: raise
`WriteOnceViolation` without touching the file.

### Integrity verification extension

`Sentinel.verify_integrity(trace_id)` already re-computes and compares
`inputs_hash` / `output_hash`. For write-once-stored traces we
additionally check:

- The on-disk file still exists.
- If the trace's `storage_mode == "writeonce_fs"` and the OS-level
  immutable flag is supported by the platform, the flag is still
  set. If the flag has been cleared since write, the result is
  `verified=False` with a `detail` explaining the discrepancy —
  "the trace's content still hashes correctly, but its write-once
  discipline has been removed".

### Cross-platform immutable flag

- **macOS**: `chflags uchg <path>` — user-immutable flag.
- **Linux**: `chattr +i <path>` — works on ext2/3/4, xfs, btrfs, zfs.
- **Windows**: no direct equivalent; skip with a debug-level log.
- **Unsupported filesystems (tmpfs, some network mounts)**: silently
  skip. The software-level write-once guarantee still holds.

Implementation uses subprocess because no stdlib call wraps these.
Failure to apply is warned, not fatal — the software rejection at
`save()` is the primary guarantee.

### Constructor

```python
from sentinel.storage.writeonce_filesystem import WriteOnceFilesystemStorage

storage = WriteOnceFilesystemStorage(path="/var/sentinel/evidence")
sentinel = Sentinel(storage=storage)
```

## Testing strategy

`tests/test_writeonce_storage.py`:

1. First save of a trace succeeds; second save of the same trace_id
   raises `WriteOnceViolation`.
2. Reading back a write-once trace works exactly like regular
   filesystem storage.
3. Querying returns all traces including write-once ones.
4. `storage_mode` field is populated on saved traces and persists
   through serialization round-trip.
5. OS-immutable flag applied on macOS / Linux (`chflags uchg` /
   `chattr +i`) when available — platform-skip-if-not-supported.
6. Integrity verification: modifying an unlocked file fails the
   verifier; verifier's own read path still works.
7. Air-gap safe: no network calls.

Reality test (fresh venv, Phase 9):

```python
from sentinel.storage.writeonce_filesystem import WriteOnceFilesystemStorage
from sentinel import Sentinel

storage = WriteOnceFilesystemStorage(path='/tmp/sentinel-wo-test')
s = Sentinel(storage=storage, signer=None)

@s.trace
def decide(x): return {'r': x}

decide('first')

# First save landed; second save of the *same* trace_id must fail.
from sentinel.storage.writeonce_filesystem import WriteOnceViolation
try:
    storage.save(s.query()[0])   # overwrite attempt
except WriteOnceViolation:
    print('write-once correctly rejected overwrite')
else:
    raise AssertionError('write-once allowed an overwrite — FAIL')
```

## Alternatives considered

**A. Enforce write-once inside `StorageBackend.save` with a flag on
`StorageBackend`.** Rejected: makes write-once a universal concern for
every backend, including those where it has no meaning (in-memory
test storage). Opt-in via a distinct class is cleaner.

**B. Use SQLite triggers for write-once.** Rejected for v3.5: SQLite
triggers are per-file, fragile under schema migration, and don't
transfer to other SQL engines. The SQLite backend stays
append-friendly; write-once lives in a dedicated backend.

**C. Rely solely on OS immutable flags.** Rejected: macOS `uchg` is
clearable with a single `chflags nouchg` without root. The
software-level rejection in `save()` is the primary guarantee; OS
flags are defense in depth.

## Rollout

- Ships in v3.5.0.
- New storage backend, opt-in; zero impact on v3.4.x callers who use
  `SQLiteStorage` or the default.
- Evidence-pack exports (PDF, NDJSON, JSON-LD) all now emit
  `storage_mode` per trace.
