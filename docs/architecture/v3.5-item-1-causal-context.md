# v3.5 Item 1 — Causal Context (OpenTelemetry Bridge)

**Design-partner architectural point 1:**
Sentinel must preserve parent-child causal relationships across agent
decisions so that a regulator — or an internal incident-review team — can
reconstruct *which decision was caused by which*, across system
boundaries.

**Status:** Accepted · 2026-04-22 · implementation in `v3.5-architecture-release` branch

---

## Problem

A single regulated AI workflow is almost never a single decision. A
credit-approval agent calls a KYC sub-agent which calls a sanctions-screen
evaluator which consults a model-scoring service. v3.4 captures every
individual decision as a trace, but the *causal chain* across those
decisions is only preserved inside Sentinel (`parent_trace_id`,
`precedent_trace_ids`) — never for the upstream world outside Sentinel
that triggered the chain.

The industry convention for cross-system causal context is **W3C Trace
Context** via OpenTelemetry. Most enterprise stacks already propagate
OTEL trace/span IDs across services for performance observability. If
Sentinel captures those IDs at trace time, every Sentinel decision
becomes joinable to the broader distributed-tracing picture — without
requiring callers to do anything.

---

## Scope — what this ADR changes

**In scope (v3.5.0 Item 1):**

- Capture the currently active W3C Trace Context at the moment a
  `@sentinel.trace` decorated function executes, if an OpenTelemetry
  SDK is installed and a span is active.
- Record three new optional fields on `DecisionTrace`:
  `otel_trace_id`, `otel_span_id`, `otel_parent_span_id`.
- Existing `parent_trace_id` and `precedent_trace_ids` remain untouched.
- No behavior change when OTEL is not installed or no span is active.

**Explicitly not in scope:**

- Becoming "OTEL-native" or re-architecting Sentinel around OTEL. Per
  `CLAUDE.md`: *"OTEL-native (Sentinel is not an OTEL-native tool;
  v3.5 provides a context **bridge**, not native status)."*
- Emitting OTEL spans from Sentinel traces. `OTelExporter` in
  `sentinel/integrations/otel.py` already handles the egress direction
  and is unchanged by this work.
- Propagating Sentinel trace IDs *into* an OTEL context for downstream
  services to consume. That's symmetry for v3.6+ if design partners ask.

## Design

### Module location

`sentinel/core/otel_context.py` — a small new module. The bridge is
part of **trace capture**, so it belongs next to the tracer, not in
`sentinel/integrations/` (which is for egress to external systems) and
not in a new `sentinel/bridges/` subpackage (one-file bridges don't
need a subpackage). Senior-engineer bar: avoid hierarchy that isn't
load-bearing.

### Public surface

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class OtelContext:
    """The three W3C Trace Context identifiers we capture.

    All strings are hex-encoded (32 chars for trace_id, 16 chars for
    span_id / parent_span_id) per W3C traceparent spec. Immutable so
    the captured context can't drift mid-trace.
    """

    trace_id: str
    span_id: str
    parent_span_id: str | None


def capture_current_otel_context() -> OtelContext | None:
    """Return the active OTEL span's context, or None.

    Returns None when:
    - ``opentelemetry-api`` is not installed (the library stays
      dependency-free by default)
    - No span is currently active
    - The active span is ``INVALID_SPAN`` (OTEL's no-op sentinel)

    Never raises. A trace that can't capture OTEL context still
    succeeds — the absence of a cross-system identifier is not a
    failure mode.
    """
```

### Soft dependency on opentelemetry-api

`opentelemetry-api` is **not** added to core dependencies. It is
optional — users who already have OTEL in their stack (which is the
entire target use-case) get the capture for free; users who don't get
``None`` returned and no behavior change. This matches the pattern used
for integrations like LangChain, Haystack, AutoGen.

The existing `[otel]` extra already pulls the SDK+exporter. An
installation of `opentelemetry-api` alone (no SDK) is also respected —
that's the common shape in agent libraries that emit spans but don't
want an opinionated SDK.

### DecisionTrace schema changes

Three new fields on `sentinel.core.trace.DecisionTrace`:

```python
otel_trace_id: str | None = None
otel_span_id: str | None = None
otel_parent_span_id: str | None = None
```

All optional with ``None`` default. Old traces without these fields
deserialize correctly — this is an **additive** schema change per
ADR-003 (no RFC required, no version bump required, optional fields
only).

### Tracer integration

`Sentinel._execute_traced` calls `capture_current_otel_context()`
once, at trace-construction time, and populates the three new fields
on the `DecisionTrace` before any policy evaluation or function
execution. Position chosen so that:

- The context captured reflects the caller's state, not anything
  Sentinel does internally
- Capture happens even if the function raises — the trace still
  records the OTEL IDs so a failing decision stays linkable
- No new async entry point required; the existing sync capture happens
  before `await func(...)` so the active span in OTEL's context var is
  whatever the caller set up.

### Precedence when both native and OTEL linkage present

Sentinel-native `parent_trace_id` continues to govern chain verification
and the `sentinel.chain` namespace/hash logic — this is a deliberate
choice: OTEL trace IDs live in the caller's telemetry backend
(Grafana, Jaeger, ...) and cannot be cryptographically verified from
within an offline Sentinel evidence pack. The OTEL fields are **joinable
identifiers for cross-system reconstruction**, not a replacement for the
hash-chain.

When reconstructing a causal chain for a regulator:

- Within Sentinel: use `parent_trace_id` → recurses to the
  sentinel-native parent.
- Across systems: use `otel_trace_id` → all Sentinel traces sharing
  that ID belong to the same distributed workflow.
- Both present: the two sets are disjoint answers to different
  questions. Sentinel-native is "who inside Sentinel triggered this
  decision"; OTEL is "which broader workflow is this part of".

Documentation in `docs/personas/` and the Phase 8 homepage refresh
will make this distinction explicit.

### What this does NOT do

- It does **not** start an OTEL span. Sentinel is a consumer of
  context, not a producer, in this direction. (The `OTelExporter`
  integration does the producer side, and is unchanged.)
- It does **not** propagate context outward. If the wrapped function
  itself makes HTTP calls or spawns subprocesses, propagating headers
  is the caller's responsibility — OTEL's own instrumentation libraries
  (requests, httpx, aiohttp, grpc) handle that, not Sentinel.
- It does **not** require changes to any existing integration. LangChain,
  Haystack, CrewAI, AutoGen, FastAPI and Django wrappers all continue to
  work; any of them running inside an OTEL-aware environment will now
  get the context on the emitted Sentinel trace for free.

## Alternatives considered

**A. OTEL-native rewrite.** Reject per canonical positioning. Sentinel
is the *evidence* layer; observability tools are the observability
layer. Peers, not replacements.

**B. Sentinel-emits-OTEL-spans bridge.** Already exists as
`sentinel.integrations.otel.OTelExporter`. Different direction, already
shipped in v3.4.

**C. Store OTEL context as a tag.** Rejected: `tags: dict[str, str]`
is the right place for opaque or domain-specific labels, not for
regulated-evidence identifiers that a verifier will explicitly query.
First-class fields are also cheaper to index in storage backends that
care about this (SQLite, Postgres).

**D. Capture OTEL context inside `DecisionTrace.__init__`**. Rejected:
couples the data class to the tracer's execution context. Keeping
capture in `_execute_traced` means `DecisionTrace` can be constructed
from NDJSON, test fixtures, or replay tools without reaching into OTEL.

## Testing strategy

All tests live in `tests/test_otel_context_bridge.py`:

1. **No OTEL installed** (monkey-patched ImportError): capture returns
   None; traces work unchanged.
2. **OTEL installed, no active span**: capture returns None.
3. **OTEL installed, active span**: capture returns a valid
   `OtelContext` with non-zero trace/span IDs.
4. **Nested spans**: capture returns the innermost (active) span's IDs;
   `parent_span_id` points at the enclosing span.
5. **OTEL `INVALID_SPAN`**: returns None (OTEL's no-op sentinel is
   treated as "no span").
6. **Tracer integration — no OTEL**: existing traces still populate as
   before, with all three new fields None.
7. **Tracer integration — OTEL active**: new trace captures the
   currently active span's IDs into `otel_trace_id` /
   `otel_span_id`.
8. **Integration — caller-span wraps Sentinel**: OTEL span created by
   caller → Sentinel-decorated function runs inside it →
   `otel_parent_span_id` on the trace equals the caller's span ID.
9. **Round-trip**: serialise-deserialise a trace with OTEL fields; all
   three fields preserved.
10. **Backward compat**: load a pre-v3.5 trace dict (missing OTEL
    fields) — deserialises cleanly, fields default None.

Coverage target: 100% of `sentinel/core/otel_context.py`.

Reality test (fresh venv, Phase 9):

```bash
pip install sentinel-kernel==3.5.0 opentelemetry-api opentelemetry-sdk
```

```python
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
trace.set_tracer_provider(TracerProvider())
tracer = trace.get_tracer("reality-test")

from sentinel import Sentinel
s = Sentinel(storage=":memory:")

@s.trace
def inner():
    return {"verdict": "approved"}

@s.trace
def outer():
    return inner()

with tracer.start_as_current_span("workflow") as parent:
    outer()

traces = s.query()
# Assert: every trace has otel_trace_id populated
# Assert: all traces share the same otel_trace_id (same workflow)
# Assert: inner trace's parent_trace_id = outer trace's trace_id
#         (sentinel-native chain preserved unchanged)
```

## Rollout

- Ships in v3.5.0 alongside items 2-4.
- Zero breaking changes.
- New fields surface automatically in evidence pack export (PDF + NDJSON).
- Added fields visible in chain attestation payload so a chain
  verifier sees OTEL identifiers in context.
