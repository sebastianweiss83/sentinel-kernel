# v3.5 Item 3 ‚Äî Fine-Grained Retention Policies

**Design-partner architectural point 3:** a single-knob
`store_inputs=True/False` on the `Sentinel()` constructor is too
coarse for regulated operators. Different agents serve different
regulatory contexts ‚Äî a credit-approval agent needs the full
GDPR Art. 6-compliant decision dossier, a customer-service chatbot
needs hash-only to minimise PII exposure, a KRITIS control-plane
agent may have jurisdictional obligations to store certain fields
and never others.

**Status:** Accepted ¬∑ 2026-04-22 ¬∑ ships in v3.5.0

---

## Problem

v3.4.x hard-codes the privacy posture at `Sentinel()` construction:
either every trace stores raw inputs/outputs or none does. A single
AI agent usually runs many `@sentinel.trace`-wrapped functions with
different regulatory contexts ‚Äî changing the constructor breaks all
of them the same way.

Regulated operators need per-decision discrimination:

- **Agent-based:** `credit_agent` stores full raw, `chat_agent` is
  hash-only.
- **Jurisdiction-based:** `data_residency="EU-DE"` retains
  differently than `data_residency="LOCAL"`.
- **Policy-family-based:** anything tagged `policy_family="BaFin"`
  has longer retention than `policy_family="internal"`.
- **Field-level redaction:** store `applicant_id` but not `ssn`.

---

## Scope

**In scope (v3.5.0 Item 3):**

- Declarative retention policy expressed as YAML. Default path
  `~/.sentinel/retention-policy.yaml`, override via
  `SENTINEL_RETENTION_POLICY` env var.
- Policy matcher runs at `_finalise_trace` ‚Äî the existing redaction
  boundary ‚Äî so policy decisions land exactly where current behaviour
  was already centralised.
- Match on: `agent` (exact or `*` wildcard suffix), `sovereign_scope`,
  `data_residency`, `tags.<key>` values.
- Actions per match: `store_inputs` (bool), `store_outputs` (bool),
  `retention_days` (advisory integer surfaced on `trace.tags`),
  `redact_fields` (list of JSONPath-style dotted paths to strip).
- First-match wins. Default rule (no match) retains v3.4.x behaviour:
  hash-only unless `store_inputs=True` / `store_outputs=True` was
  passed to `Sentinel()`.

**Explicitly not in scope:**

- Automatic deletion of traces past `retention_days`. v3.5 captures
  the retention window; enforcement (cron-style sweeper) is v3.6.
- Full GDPR Art. 17 deletion workflow. The `sentinel purge` CLI
  already exists; v3.5 adds retention metadata, v3.6 wires it in.
- Complex JSONPath filters with predicates (`$.foo[?(@.x>1)]`). v3.5
  uses simple dotted-key paths (`foo.bar.ssn`). Enough for the target
  use-cases; no external dep.
- Per-user retention policies. The policy is process-wide in v3.5.

## Design

### File format

```yaml
# ~/.sentinel/retention-policy.yaml
version: 1
rules:
  - name: credit-agents-full-retention
    match:
      agent: credit_*
      sovereign_scope: EU
    actions:
      store_inputs: true
      store_outputs: true
      retention_days: 3650

  - name: chat-agents-hash-only
    match:
      agent: chat_*
    actions:
      store_inputs: false
      store_outputs: false
      retention_days: 30

  - name: bafin-policy-family
    match:
      tags:
        policy_family: BaFin
    actions:
      store_inputs: true
      store_outputs: true
      retention_days: 3650
      redact_fields:
        - inputs.ssn
        - inputs.iban
        - output.credit_score_raw
```

Top-level `version: 1` is mandatory ‚Äî future schema changes bump it
so a v2-file can't be silently misinterpreted by a v1-reader.

### Matching semantics

Every rule is tested in file order. First full match wins. A rule
with no `match:` clause matches everything (valid for the last rule
as a catch-all).

Within a `match:` block:

- `agent` accepts exact names or a single trailing `*` wildcard
  (`credit_*` matches `credit_agent`, `credit_decision`). No regex,
  no multi-char wildcards ‚Äî keep YAML human-readable.
- `sovereign_scope`, `data_residency` match literally.
- `tags.<key>: value` matches when `trace.tags[<key>] == value`.
  Missing tag ‚Üí no match for that rule.

### Action application

Applied at `_finalise_trace` (the redaction boundary):

1. `store_inputs: false` ‚Üí `trace.inputs = {}`
2. `store_outputs: false` ‚Üí `trace.output = {}`
3. `redact_fields` list ‚Üí walk each dotted path, delete leaf keys.
   Inputs/outputs hashes (`inputs_hash`, `output_hash`) are computed
   **before** redaction, so integrity verification still works against
   the original full payload.
4. `retention_days` ‚Üí surface on `trace.tags["retention_days"]` as a
   stringified int. Enforcement is out of scope; capturing the
   intended window is in scope.

### Integration with existing `Sentinel(store_inputs=...)` kwargs

The constructor kwargs and the YAML policy layer cleanly:

- If YAML defines a matching rule with `store_inputs` explicit ‚Üí YAML
  wins.
- If no YAML rule matches ‚Üí v3.4.x behaviour (constructor kwargs
  decide).
- YAML file absent ‚Üí v3.4.x behaviour (constructor kwargs decide).

No behaviour change for operators who never create the YAML file.

### Loading and caching

The policy is loaded once per `Sentinel()` instance from:

1. `SENTINEL_RETENTION_POLICY` env var ‚Üí file at that path.
2. `~/.sentinel/retention-policy.yaml`.
3. Fallback: no policy, constructor kwargs govern.

Load failures (malformed YAML, unknown `version`, unknown action
keys) raise a clear `ValueError` at construction time ‚Äî fail fast,
not silently.

### Schema validation

Strict unknown-key rejection. Unknown top-level keys, unknown rule
keys, unknown match keys, unknown action keys all raise. Typos in
policy files should never silently no-op.

## Alternatives considered

**A. Policies as Python callables passed to `Sentinel()`.** Rejected:
not operator-friendly. Regulated-ops teams ship YAML, not Python.

**B. Dynamic policy loading per decision.** Rejected: a per-request
YAML re-read is a performance and consistency risk. If operators need
runtime policy rotation they can restart the worker.

**C. Complex expression language (OPA Rego).** Rejected for v3.5:
OPA is already integrated as the policy *evaluator* for ALLOW/DENY;
adding it for retention confuses concerns. Simple YAML match is
sufficient for the retention problem.

**D. Store policy inside the database rather than a file.** Rejected:
the file is reviewable in a PR and version-controllable by the
operator's config-management tool; a DB-stored policy isn't.

## Testing strategy

`tests/test_retention_policy.py`:

1. No policy file ‚Üí v3.4.x behaviour preserved (constructor kwargs
   decide).
2. YAML file with matching rule ‚Üí overrides constructor.
3. Agent-wildcard match (`credit_*`) ‚Äî matches credit_agent,
   credit_decision, etc.
4. Tag-based match (`tags.policy_family: BaFin`) ‚Äî non-BaFin traces
   fall through to default.
5. `redact_fields` strips listed dotted paths from stored
   inputs/outputs **after** the hashes are computed, so
   `verify_integrity` still succeeds against the redacted payload
   when re-hashed from retained fields. (Integrity against original
   PII payload intentionally impossible ‚Äî that's the point of
   redaction.)
6. `retention_days` surfaces on `trace.tags["retention_days"]`.
7. Unknown `version:` raises `ValueError`.
8. Unknown action / match key raises `ValueError`.
9. First-match-wins: two overlapping rules ‚Üí first rule's actions
   applied.
10. Air-gap-friendly: no network, no external services.

Reality test (fresh venv, Phase 9):

```bash
pip install 'sentinel-kernel[jsonld]==3.5.0' pyyaml
```

```python
import os, tempfile, textwrap
yaml_policy = textwrap.dedent('''
version: 1
rules:
  - match: { agent: credit_* }
    actions: { store_inputs: true, store_outputs: true, retention_days: 3650 }
  - match: { agent: chat_* }
    actions: { store_inputs: false, store_outputs: false, retention_days: 30 }
''')
with tempfile.NamedTemporaryFile('w', suffix='.yaml', delete=False) as f:
    f.write(yaml_policy)
os.environ['SENTINEL_RETENTION_POLICY'] = f.name

from sentinel import Sentinel
s = Sentinel(storage=':memory:', signer=None)

@s.trace(agent_name='credit_agent')
def credit(ssn): return {'approved': True, 'ssn': ssn}

@s.trace(agent_name='chat_agent')
def chat(msg): return {'reply': 'hi'}

credit(ssn='123-45-6789')
chat(msg='hello')

# Credit trace retains raw input/output + 10-year retention window.
# Chat trace stores hashes only + 30-day window.
traces = {t.agent: t for t in s.query()}
assert traces['credit_agent'].inputs != {}
assert traces['credit_agent'].tags['retention_days'] == '3650'
assert traces['chat_agent'].inputs == {}
assert traces['chat_agent'].tags['retention_days'] == '30'
```
